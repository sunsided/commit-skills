# Triaging review comments

Triage has one job: for each comment, decide between **fix-and-reply** and **reply-only** — and never let a comment fall into *do-nothing-and-stay-silent*. The labels below are a thinking aid, not a taxonomy to get right. A comment can be two things at once (a question that's really a defect); when in doubt, lean toward the action that closes the loop most honestly.

The thread is also a record other people read later. Your reply should make sense to someone scrolling the PR in six months who never saw the diff.

## The kinds

### Defect — fix and reply

A real problem: a bug, an off-by-one, an unhandled `None`, a race, a missing validation, a security or performance hole. The reviewer is correct and the code is wrong.

Action: fix it, commit, reply with what changed + the SHA.

> **Comment:** "This expiry check uses `>`, so a token expiring exactly now is still served once."
> **Reply:** Fixed in 4f2a1c8. It's `>=` now, so a token expiring on the boundary tick counts as expired.

Lead with `Fixed in <sha>.` so a reviewer skimming the thread reads the outcome before the rationale. Then don't over-explain — the diff shows the change; the reply says what and confirms you understood *why*.

### Suggestion — fix if you agree, else reason back

A reasonable improvement that isn't strictly necessary: extract a helper, rename for clarity, simplify a branch, add a test. Often prefixed "nit:". Judge it on merit and cost.

If you agree and it's cheap:

> **Comment:** "nit: could pull these three lines into a `normalize_path` helper — they repeat in two other places."
> **Reply:** Fixed in 9b1e0a2. Extracted `normalize_path` and routed all three call sites through it.

If you disagree, say why — a reasoned decline is a perfectly good outcome:

> **Comment:** "Consider a list comprehension here instead of the loop."
> **Reply:** I'd keep the explicit loop — it builds the result lazily and this runs per-request in the hot path; the comprehension would materialize the whole list each call. Happy to switch if you've measured it as a non-issue.

### Opinion — usually reply-only

Pure taste with no clear winner: naming flavor, where to put a blank line, prose in a doc comment. Fixing is fine if it's cheap and you don't actively prefer your version; otherwise acknowledge the tradeoff and move on. Don't make the code worse to satisfy a preference, and don't argue taste at length.

> **Comment:** "I'd call this `fetchUser` rather than `loadUser`."
> **Reply (conceding):** Fixed in c3d9f00. Renamed to `fetchUser`.
> **Reply (holding):** I lean `loadUser` for consistency with `loadSession`/`loadConfig` next to it — keeping it, but no strong feelings.

### Question — answer, and fix if it exposes a gap

The reviewer wants to understand something. Sometimes it's pure curiosity; sometimes "what happens if this is empty?" is a defect report in disguise. Answer directly. If digging in reveals a real hole, fix it and say so.

> **Comment:** "What happens if `items` is empty here?"
> **Reply (it's fine):** Empty is handled — the loop doesn't execute and we return the zero-value accumulator, covered by `test_empty_batch`.
> **Reply (it wasn't fine):** Fixed in 1a2b3c4. Good question — it would have divided by zero; added an early return for the empty case.

### Praise / non-actionable — no reply

"👍", "nice refactor", "TIL". Resist replying to every one; an empty thread is better than a wall of "thanks!". Reply only if there's something real to add.

## Judgment calls

- **A pile of nits from one reviewer.** Batch the fixes into one or a few commits, but still reply in each thread (a one-liner citing the shared SHA is fine). The reviewer opened each thread; each deserves a close.
- **Reviewer is factually wrong.** Don't debate — point at the line or test that shows the real behavior, neutrally. "`get()` already takes the lock first — see line 41." Let the code make the argument.
- **Suggestion is good but out of scope.** Reply proposing a follow-up issue rather than growing the diff; file and link it if the user agrees. "Agree this whole module wants the same treatment — that's bigger than this PR, filed #418 to track it."
- **You can't tell what they mean.** Ask. A specific clarifying question beats guessing and fixing the wrong thing.
- **Conflicting comments from two reviewers.** Surface the conflict rather than silently picking a side: reply noting both, propose a resolution, let them settle it.

## What every reply avoids

- Tooling chatter — "I ran the tests and they pass", "generated by", AI footers.
- Reflexive gratitude on every thread. Genuine once or twice; not twelve times.
- Re-describing the diff in prose. Cite the SHA and let the change speak.
- Backticks around the SHA — keep it bare so GitHub links it (see SKILL.md).
- Caving on a change you think is wrong just to close the thread. Reply-only is a valid, respected outcome.
